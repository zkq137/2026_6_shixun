# 需求规格说明书

## 1. 项目背景

传统轻量电商项目通常只覆盖商品、购物车和订单流程，缺少智能化亮点。本项目在电商基础业务之上，引入多 Agent 与工具调用能力，让系统可以根据用户行为推荐商品，回答客服问题，辅助管理员分析销售情况，并预测库存风险。

## 2. 项目目标

- 建立一个可演示、可扩展的轻量智能商城。
- 支持用户端完整购物闭环。
- 支持管理员对商品、订单、库存和用户进行管理。
- 通过用户行为数据驱动推荐、运营分析和库存预测。
- 用多 Agent 展示 AI 调用工具完成业务任务的过程。

## 3. 用户角色

| 角色 | 说明 | 主要能力 |
| --- | --- | --- |
| 游客 | 未登录用户 | 浏览商品、搜索商品、查看商品详情 |
| 普通用户 | 已登录买家 | 加购、下单、查看订单、使用客服和推荐 |
| 管理员 | 后台管理人员 | 管理商品、分类、订单、库存、用户和 AI 分析 |
| AI Agent | 系统智能助手 | 通过工具查询业务数据并生成回答或建议 |

## 4. 功能需求

### 4.1 用户认证

- 用户可以注册账号。
- 用户可以使用账号密码登录。
- 用户登录后获得访问令牌。
- 用户可以查看和修改个人基础信息。
- 管理员通过独立后台入口登录。

优先级：MVP。

验收标准：

- 注册时校验手机号或用户名唯一。
- 登录成功返回 token。
- 未登录用户不能访问购物车、订单和个人中心。

### 4.2 商品与分类

- 首页展示推荐商品、热门商品和分类入口。
- 商品支持按分类、关键词、价格区间、上下架状态筛选。
- 商品详情展示名称、图片、价格、库存、销量、描述和推荐区。
- 管理员可以新增、编辑、上下架、删除商品。
- 管理员可以维护商品分类。

优先级：MVP。

验收标准：

- 用户可以正常浏览商品列表和详情。
- 搜索结果与关键词、分类条件匹配。
- 下架商品不在用户端展示。
- 后台商品修改后，用户端展示同步变化。

### 4.3 购物车

- 用户可以将商品加入购物车。
- 用户可以修改购物车商品数量。
- 用户可以删除购物车商品。
- 购物车展示商品单价、数量、小计和总价。
- 库存不足时禁止提交超量商品。

优先级：MVP。

验收标准：

- 同一用户同一商品重复加入时累加数量。
- 商品下架或库存不足时给出明确提示。
- 购物车金额计算准确。

### 4.4 订单

- 用户可以从购物车提交订单。
- 订单生成后扣减库存。
- 用户可以查看自己的订单列表和订单详情。
- 用户可以取消未发货订单。
- 管理员可以查看订单并修改订单状态。

订单状态：

```text
pending 待处理
paid 已支付或已确认
shipped 已发货
completed 已完成
cancelled 已取消
```

优先级：MVP。

验收标准：

- 提交订单时生成订单主表和订单明细。
- 订单总价等于明细商品价格乘数量之和。
- 取消订单后恢复库存。
- 用户不能查看其他用户订单。

### 4.5 后台管理

- 后台首页展示销售额、订单数、用户数、库存风险数量。
- 管理员可以管理商品、分类、订单、库存和用户。
- 管理员可以查看 Agent 工具调用日志。
- 管理员可以查看销量统计、预测结果和库存预警。

优先级：重要。

验收标准：

- 后台操作需要管理员权限。
- 商品和订单管理支持分页、搜索和状态筛选。
- AI 分析结果能从后台入口查看。

## 5. AI 功能需求

### 5.1 客服 Agent

职责：

- 回答退换货、物流、售后政策、订单状态等问题。
- 优先查询 FAQ。
- FAQ 无法回答时调用大模型生成回答。
- 需要订单信息时调用订单查询工具。

优先级：重要。

验收标准：

- 常见问题能命中 FAQ。
- 用户询问订单状态时，Agent 能调用 `get_user_orders` 或 `get_order_detail`。
- 大模型不可用时返回兜底提示。

### 5.2 推荐 Agent

职责：

- 根据浏览、加购、购买行为生成个性化推荐。
- 支持热门推荐、看了又看、买了又买。
- 为推荐结果生成简短理由。

优先级：重要。

验收标准：

- 未登录用户返回热门商品。
- 登录用户优先基于用户行为推荐。
- 推荐结果不包含下架商品。

### 5.3 运营 Agent

职责：

- 分析热销商品、滞销商品、销售趋势和用户行为。
- 回答管理员的自然语言经营问题。
- 生成促销、补货、下架或优化建议。

优先级：重要。

验收标准：

- 管理员可以提问“最近哪些商品销量最好”。
- Agent 能调用销售统计工具并返回结构化结论。
- 返回内容包含数据依据和建议。

### 5.4 库存 Agent

职责：

- 根据库存、历史销量和预测销量判断库存风险。
- 给出补货建议。
- 可生成库存预警记录。

优先级：重要。

验收标准：

- 能识别库存不足风险。
- 能说明预测依据。
- 能调用 `create_inventory_alert` 生成预警。

## 6. 非功能需求

### 6.1 性能

- 商品列表接口在常规数据量下响应时间不超过 1 秒。
- 搜索、分页、后台列表接口需要使用索引字段。
- 推荐结果允许缓存，避免每次实时计算。

### 6.2 安全

- 用户和管理员接口必须区分权限。
- 密码必须加密存储。
- 用户只能访问自己的购物车和订单。
- Agent 工具调用必须进行权限校验。
- 大模型 Prompt 不应暴露数据库连接、密钥和内部系统信息。

### 6.3 可维护性

- 后端采用 API、Service、Repository、Model 分层。
- Agent 通过工具调用业务能力，不直接访问数据库。
- 配置项通过环境变量管理。
- 文档中的接口、表名、工具名作为开发基准。

### 6.4 可用性

- 大模型服务不可用时，客服功能应能通过 FAQ 降级。
- Redis 不可用时，核心购物流程仍应可用。
- 推荐失败时返回热门商品兜底。

## 7. 功能优先级

| 优先级 | 功能 |
| --- | --- |
| MVP | 登录注册、商品、分类、购物车、订单、后台商品管理 |
| 重要 | 用户行为记录、推荐、客服 Agent、后台订单管理、销量统计 |
| 增强 | 运营 Agent、库存 Agent、预测、预警、Agent 调用日志 |
| 可选 | 支付模拟、优惠券、评论、收藏、多轮复杂任务 |

## 8. 总体验收标准

- 用户端可以完成从浏览商品到提交订单的闭环。
- 管理员可以在后台维护商品、分类、订单和库存。
- 系统能记录用户行为，并基于行为生成推荐。
- 至少 4 个 Agent 的职责清晰，并能通过工具调用完成任务。
- 后台能查看销量预测、库存预警和 Agent 工具调用日志。
- 所有核心功能有对应接口文档、数据库表设计和测试场景。


## 2026-06-23 更新：商品图片管理需求

后台商品管理补充以下能力：

- 管理员可直接上传商品主图。
- 管理员可预览、替换、清空商品主图。
- 上传图片保存到服务器本地，商品表 `main_image` 保存访问路径。
- 用户端商品卡片和商品详情页需要展示商品主图，图片失效时显示兜底占位。
